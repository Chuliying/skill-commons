# 產文風格規範

適用範圍：skills 產出的所有 durable Markdown artifact（PRD、Spec、plan、report、README 等）
與本 repo 自身的文件。SKILL.md 指示文與 repo 文件同受約束；vendored 外部技能的 upstream
原文不改寫（更新時整段保留）。

目的：文件是交接契約與長期資產，風格不一致本身就是熵。本規範收斂 LLM 產文的常見缺陷，
讓任何 agent 在任何 session 產出的文件讀起來像同一個作者。

## 規則

### 1. 禁用 emoji

文件內文、標題、表格、狀態標記一律不用 emoji。狀態以文字表達：

| 原寫法 | 改為 |
|---|---|
| ✅ 完成 | PASS 或「完成」 |
| ❌ 失敗 | FAIL 或「失敗」 |
| ⚠️ 注意 | 「注意：」 |
| 🚦 Gate | Gate |

例外：無。既有檔案的 emoji 依 public-release-plan 的遷移任務分批移除。

### 2. 禁用對比修辭模板

以下句式是 LLM 慣性，直接陳述主張即可：

- 「不是 X，而是 Y」「並非 X，而是 Y」——直接寫 Y，必要時補一句為何排除 X。
- 「不只是 X，更是 Y」「X 不僅僅是…」
- 「與其說 X，不如說 Y」
- 「這不僅提升了…也…」

判斷標準：刪掉前半句後語意不變，就該刪。

### 3. 用語採台灣慣用詞

| 禁用 | 改為 | 禁用 | 改為 |
|---|---|---|---|
| 質量 | 品質 | 信息 | 資訊 |
| 軟件 | 軟體 | 硬件 | 硬體 |
| 網絡 | 網路 | 視頻 | 影片 |
| 服務器 | 伺服器 | 數據庫 | 資料庫 |
| 代碼 | 程式碼 | 調用 | 呼叫 |
| 對象（object） | 物件 | 缺省 | 預設 |
| 交互 | 互動 | 組件 | 元件 |
| 文檔 | 文件 | 屏幕 | 螢幕 |
| 內存 | 記憶體 | 變量 | 變數 |
| 函數 | 函式 | 接口 | 介面 |
| 循環 | 迴圈 | 支持（support） | 支援 |
| 項目（project） | 專案 | 打印 | 列印 |

語境例外：「通過測試」的「通過」（pass）合法；「透過」用於 via。
「項目」指清單條目時合法。引用外部文獻原文不改。

### 4. 禁用空詞與儀式性語句

- 膨脹形容：強大、完美、極大提升、全面、深度整合、賦能——主張必須附證據或刪除。
- 儀式性開場與收尾：「值得注意的是」「總而言之」「綜上所述」「讓我們」——刪除。
- 「進行」贅詞：「進行測試」寫「測試」。

### 5. 標點與排版

- 中文內文用全形標點（，。：；）；程式碼、路徑、指令、識別字用反引號與半形。
- 中文與英文、中文與數字之間留一個半形空格。
- 標題層級不跳號（h2 之下不直接出現 h4）。
- 表格只放可枚舉的事實；解釋與推理寫在段落，不塞進儲存格。

### 6. 結構紀律

- 一句話一個主張；一段落一個主題。
- 主張在前，論據在後；文件開頭三行內能回答「這份文件是什麼、給誰看」。
- 不建立只有一句話內容的小節；併入上層段落。

## 強制方式

可機械判定的規則（emoji、第 3 條 denylist、「不是…而是」正則）由
`scripts/lint-docs.sh` 檢查並掛入測試（見 public-release-plan T2-3）；
其餘規則由 review 階段把關。新增文件即受檢；既有檔案依遷移任務分批修正，
遷移完成前 lint 只對新增與修改的檔案生效。
